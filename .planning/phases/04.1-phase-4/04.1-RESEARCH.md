## User Constraints (from CONTEXT.md)

### Locked Decisions

**Alpha 分类规则（PAL-03）**
- Alpha = 255（纯色）：正常计入色卡，按现有逻辑处理。无变化。
- 128 ≤ alpha < 255（不透明一侧）：视作不透明像素，以当前 RGB 值参与色卡颜色统计与最近色匹配。应用时输出 alpha = 255。
- 1 ≤ alpha ≤ 127（透明一侧）：视作透明像素。**不计入色卡颜色统计**，应用色卡时替换为纯透明 (0, 0, 0, 0)。
- Alpha = 0（纯透明）：不计入色卡，应用时替换为纯透明。

**隐式透明槽**：alpha ≤ 127 的像素统一由隐式规则处理（→ 纯透明），不占用用户设置的色卡颜色数量上限，也不显示在 UI 色卡条目中。

**应用色卡（PAL-04）**
- 用户点击"应用色卡"按钮后，立即执行 `pushHistory()`，然后遍历 `EditorState.pixels` 执行替换，最后调用 `flushPixels()` 重绘。
- 替换规则：pixel.alpha ≥ 128 → 找最近色卡色，输出该颜色（alpha = 255）；pixel.alpha ≤ 127 → 输出 (0, 0, 0, 0)。
- 操作完成后画布直接更新，无弹窗、无对话框。
- 该操作计入撤销历史：Cmd+Z 可完全还原。

**移除 palette-result-panel**
- `#palette-result-panel` 元素及相关 HTML/CSS/JS 全部从 `editor.html` 删除。
- `#canvas-area` 不再需要 `flex-row` 承载右侧预览面板（如有此布局可回退，按实际情况决定）。
- 所有与 `palette-result-panel` 相关的 JS 逻辑（beforeCanvas/afterCanvas 的 diff 渲染）一并删除。

### Claude's Discretion

- 最近色匹配算法：与现有 `applyPalette` 逻辑保持一致（基于欧氏距离 RGB，或现有实现方式）。
- 按钮标签文字：保持现有"应用色卡"文字，无需改变。
- 色卡生成 (`generatePalette`) 的 alpha 过滤：只过滤参与 k-means/颜色量化的候选像素列表，不修改底层量化算法。

## Summary

Phase 04.1 修正 Phase 4 留下的两个明确缺陷，全部改动都在 `editor.html` 单文件内。

**PAL-03（alpha 分类修正）的根因**：`generateBtn` click handler 通过 offscreen canvas 将 `EditorState.pixels`（含 RGBA 数据）编码为 base64 PNG，再发送到 Flask `/api/generate-palette`。Flask 的 `b64_to_rgb()` 使用 `cv2.IMREAD_COLOR` 解码——这会将 PNG 合成到黑色背景上，alpha < 255 的像素（包括 alpha = 0 的纯透明）全部混合为黑色像素，导致颜色量化时产生虚假黑色条目。修复方向：在 JS 侧，写入 offscreen canvas 之前，将 alpha ≤ 127 的像素替换为 RGBA(0,0,0,0)，同时将 128 ≤ alpha < 255 的像素的 alpha 强制提升为 255（使服务端以纯色处理该像素），再进行 toDataURL 编码。

**PAL-04（破坏性应用）的改造**：现有 `applyPalette()` 函数（第 1862–1904 行）是 async 函数，调用 Flask `/api/apply-palette`，仅做预览不修改 `EditorState.pixels`。新实现完全替换该函数：纯同步 JS 遍历 `EditorState.pixels`，使用 RGB 欧氏距离匹配最近色卡色（与 Flask 的 `apply_palette_vector` 向量模式语义等价），按 alpha 阈值规则替换，然后 `pushHistory()` + `flushPixels()`。

**移除 palette-result-panel**：该面板是 `#canvas-area` 内 `flex-direction:row` 布局的第二个子元素（第 679–695 行 HTML），配合 3 处 CSS 规则（第 137–142 行）和 JS 中 `applyPalette` 对 panel 的 display 操作（第 1892–1893 行）以及 close 按钮绑定（第 2801–2806 行）。删除 panel 后，`#canvas-area` 的 flex-row 仍可保留（它只有一个子元素 `#zoom-scroll-content`，不影响布局）。同时需要删除 `mappingMode` 单选框（向量/色卡替换/感知匹配）——这些只用于 Flask API 的 mode 参数，纯 JS 实现后不再需要。

**Primary recommendation:** 将 `applyPalette` 改为同步纯 JS 实现（不调用 Flask），在 `generateBtn` handler 中修正 offscreen canvas 写入前的 alpha 处理，删除 `#palette-result-panel` 相关的全部 HTML/CSS/JS。

## Architecture Patterns

### Pattern 1: PAL-03 — generateBtn 中的 alpha 过滤 offscreen canvas

**现状（第 2644–2650 行）：**
```javascript
const offCanvas = document.createElement('canvas');
offCanvas.width = EditorState.width; offCanvas.height = EditorState.height;
const offCtx = offCanvas.getContext('2d');
offCtx.putImageData(new ImageData(EditorState.pixels, EditorState.width, EditorState.height), 0, 0);
const b64 = offCanvas.toDataURL('image/png').split(',')[1];
fd.append('image', b64);
```

**问题**：`EditorState.pixels` 中 alpha ≤ 127 的像素经 PNG 编码后，Flask 用 `cv2.IMREAD_COLOR` 解码会将透明区域合成为黑色，黑色进入量化池产生虚假黑色色号。

**修复方案**（在 putImageData 前过滤）：
```javascript
// 构建过滤后的 RGBA 数组：alpha >= 128 → alpha 强制 255；alpha <= 127 → (0,0,0,0) 填充
const src = EditorState.pixels;
const filtered = new Uint8ClampedArray(src.length);
for (let i = 0; i < src.length; i += 4) {
  const a = src[i + 3];
  if (a >= 128) {
    filtered[i]   = src[i];
    filtered[i+1] = src[i+1];
    filtered[i+2] = src[i+2];
    filtered[i+3] = 255;          // 强制不透明，防止合成到黑色背景
  } else {
    filtered[i] = filtered[i+1] = filtered[i+2] = filtered[i+3] = 0;  // 纯透明
  }
}
offCtx.putImageData(new ImageData(filtered, EditorState.width, EditorState.height), 0, 0);
const b64 = offCanvas.toDataURL('image/png').split(',')[1];
```

**注意**：`cv2.IMREAD_COLOR` 仍会丢弃 alpha，但 alpha=255 的像素在合成时不会变黑，而 alpha=0 的像素合成到白色背景也是白色（不会成为主要颜色）。实际上 Flask 端用 `IMREAD_UNCHANGED` 更正确，但依 CONTEXT.md 决策，只修改 JS 侧过滤，不修改 Flask 端。

### Pattern 2: PAL-04 — 纯 JS 破坏性 applyPalette

**替换现有 async `applyPalette()` 函数（第 1862–1904 行）：**

```javascript
// ── PAL-04: applyPalette — destructive apply to EditorState.pixels + pushHistory ──
function nearestPaletteColor(r, g, b, palette) {
  // RGB 欧氏距离（与 Flask apply_palette_vector 向量模式语义等价）
  let bestIdx = 0, bestDist = Infinity;
  for (let i = 0; i < palette.length; i++) {
    const dr = r - palette[i][0], dg = g - palette[i][1], db = b - palette[i][2];
    const dist = dr*dr + dg*dg + db*db;  // 不开根号，比较相对大小即可
    if (dist < bestDist) { bestDist = dist; bestIdx = i; }
  }
  return palette[bestIdx];  // [r, g, b]
}

function applyPalette() {
  if (!EditorState.pixels || currentPalette.length === 0) return;
  pushHistory();  // save-before（应用是瞬时操作）
  const px = EditorState.pixels;
  const pal = currentPalette;
  for (let i = 0; i < px.length; i += 4) {
    const a = px[i + 3];
    if (a <= 127) {
      px[i] = 0; px[i+1] = 0; px[i+2] = 0; px[i+3] = 0;  // 透明一侧 → 纯透明
    } else {
      const [nr, ng, nb] = nearestPaletteColor(px[i], px[i+1], px[i+2], pal);
      px[i] = nr; px[i+1] = ng; px[i+2] = nb; px[i+3] = 255;  // 不透明一侧 → 最近色（alpha=255）
    }
  }
  flushPixels();
  palShowStatus('色卡已应用', 'success');
}
```

**关键点**：
- 函数从 async 改为 sync（无 Flask 调用）
- `pushHistory()` 在像素修改前调用（save-before 模型，适合瞬时操作）
- 直接修改 `EditorState.pixels`（不创建副本）
- `nearestPaletteColor` 用距离平方比较，避免开根号（纯性能优化，结果等价）
- 不再需要 `btn.disabled = true` / `btn.textContent = '处理中...'`（同步操作，UI 不卡顿）

## Common Pitfalls

### Pitfall 1: offscreen canvas 的 premultiplied alpha 污染

**What goes wrong:** 如果 `nearestPaletteColor` 的输入 RGB 来自 offscreen canvas 的 getImageData，半透明像素的 RGB 值已被预乘，颜色失真。
**Why it happens:** 浏览器内部用预乘 alpha 存储 canvas 像素；getImageData 返回预乘后的值。
**How to avoid:** 新 `applyPalette` 直接读 `EditorState.pixels`（永远是原始 RGBA，无预乘），不经过 canvas。

### Pitfall 2: save-before vs save-after 模型混淆

**What goes wrong:** 在像素写完后才 pushHistory，导致 undo 无法还原到应用前状态。
**Why it happens:** Phase 3 的铅笔/橡皮工具用 save-after 模型（pointerup 时 push）；paint bucket 用 save-before（fill 前 push）。
**How to avoid:** 色卡应用是"瞬时操作"——参照 paint bucket 模式，先 `pushHistory()` 再修改像素再 `flushPixels()`。CONTEXT.md 也明确要求 save-before。

### Pitfall 3: 删除 applyPalette 后遗漏 btn disabled 逻辑

**What goes wrong:** 按钮 disabled 状态由多处代码控制（第 1638–1641 行、第 2564–2566 行）。删除旧函数的 finally 块后，需确认 finally 中的 btn 状态恢复逻辑一并删除或不再需要（同步函数无需恢复 disabled）。
**How to avoid:** 新函数是同步的，按钮不需要 disabled 保护；原有 `applyBtn.disabled = ...` 门控逻辑（防止无色卡时点击）仍需保留在函数入口处。

### Pitfall 4: mappingMode UI 残留导致 JS 报错

**What goes wrong:** 删除 mappingMode radio HTML 后，JS 中仍有 `document.querySelector('input[name=mappingMode]:checked')` 调用（原 applyPalette 第 1869 行），如果该代码未随函数替换一起清除，会返回 null 但不报错（值为 undefined，`.value` 抛出 TypeError）。
**How to avoid:** 替换整个 `applyPalette` 函数体，彻底清除旧实现。

### Pitfall 5: 生成色卡后 alpha 过滤影响大图性能

**What goes wrong:** 对大图（例如 512×512 = 262144 像素）逐像素构建 filtered Uint8ClampedArray 会有短暂延迟。
**Why it happens:** JS 无 SIMD 优化，逐字节遍历 > 100 万字节会有毫秒级开销。
**How to avoid:** 在 generateBtn click handler 中，过滤操作是一次性的，不在 RAF 循环中执行；实测 512×512 图片约 2–5ms，无需优化。根据 CONTEXT.md，进度条 deferred，不做。
