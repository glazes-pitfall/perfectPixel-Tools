## Summary

Phase 4.2 是纯 `editor.html` 前端重构，无 Flask 后端改动。工作量集中在三块：

1. **删除** — 移除 `#paletteEnabled` toggle HTML（第 477–480 行）及其 JS 事件监听（第 2624–2638 行）；移除 tab 结构（第 488–531 行）；移除 3 个 radio card（第 570–592 行）；移除独立导出 section（第 595–603 行）；移除 `#addColorBtn`（第 543 行）；移除 `#loadPaletteBtn`、`#deletePaletteBtn`（第 562–564 行）及其 JS 绑定（第 2826–2855 行）。

2. **重排** — 将 `#paletteBody` 内部结构由「tab-generate / tab-upload → 色卡 → 保存加载 → 映射 → 导出 → 应用」改为「色卡 → combobox → 生成区 → 固定底部（映射 + 应用）」。

所有 JS 函数（`renderSwatches`、`setCurrentPalette`、`applyPalette`、`uploadPaletteFile`、`exportPalette`、`getSavedPalettes`、`setSavedPalettes`、`refreshSavedDropdown`）逻辑不变，只改 DOM 绑定方式和读取目标元素 ID。

### Pattern 1：`.palette-body` 改为 flex column 可滚动布局

**What:** `#paletteBody` 从简单 flex column（`gap:10px`）改为「可滚动主区 + sticky 底部」
**当前 CSS（第 227 行）:**
```css
.palette-body { padding: 12px; display: flex; flex-direction: column; gap: 10px; }
```
**改为:**
```css
.palette-body {
  display: flex;
  flex-direction: column;
  overflow: hidden;      /* 让内部 .pal-scroll-area 控制滚动 */
  max-height: calc(100vh - 280px);   /* 留出 color picker + pixel inspector 高度 */
}
.pal-scroll-area {
  flex: 1;
  overflow-y: auto;
  padding: 12px;
  display: flex;
  flex-direction: column;
  gap: 10px;
}
.pal-sticky-bottom {
  flex-shrink: 0;
  padding: 10px 12px;
  border-top: 1px solid var(--border);
  background: var(--surface);
  display: flex;
  flex-direction: column;
  gap: 8px;
}
```

**或者（更简单）使用 sticky 方案:**
```css
.palette-body { padding: 12px; display: flex; flex-direction: column; gap: 10px; }
/* sticky-bottom 不需要额外父容器改动 */
.pal-sticky-bottom {
  position: sticky;
  bottom: 0;
  background: var(--surface);
  border-top: 1px solid var(--border);
  padding-top: 10px;
  margin-top: 4px;
  display: flex;
  flex-direction: column;
  gap: 8px;
}
```

**推荐 sticky 方案** — 不需要修改 `#paletteSection` 的 overflow:hidden（注意：当前 `.palette-section` 有 `overflow: hidden`（第 218 行），sticky 在 overflow:hidden 父容器内不生效，需改为 `overflow: visible` 或将 sticky 改为 flex-shrink:0 方案）。

**结论：采用 flex-shrink:0 方案（见 D7 备注），避免 overflow:hidden 与 sticky 的冲突。**

### Anti-Patterns to Avoid

- **不要** 用 `position:sticky` 在 `overflow:hidden` 父容器内（`.palette-section` 有 `overflow:hidden`，sticky 会失效）。用 flex-shrink:0 方案代替。
- **不要** 在 `renderSwatches()` 中给 `.add-swatch` 添加 `del-btn` 子元素——它不是真正的 swatch，没有索引。
- **不要** 修改 `.custom-select-options` 的 CSS position（原为 `position:absolute; top:calc(100%+4px)`）——combobox 新布局让 options div 直接跟随行内，用 `position:relative; top:0` 或放弃 absolute 改为 block 展示更干净。
- **不要** 删除 `#palFileInput`——上传逻辑（`uploadPaletteFile`）仍使用它，只是触发方式改为程序性 `.click()`。
- **不要** 遗漏 `paletteEnabled.change` 监听器的删除——第 2624–2638 行整块删除，否则引用已删除的元素（`paletteEnabled` 为 null）不影响功能但有 console 警告。
- **不要** 修改 tab-btn `.querySelectorAll` 以外的 JS（第 2641–2649 行）——tab 结构删除后这段代码会因为找不到元素而静默无操作，但最好一并删除。

## Common Pitfalls

### Pitfall 1：sticky 在 overflow:hidden 容器内失效

**What goes wrong:** `.palette-section` 的 CSS（第 218 行）有 `overflow: hidden`，导致内部的 `position:sticky` 子元素无法相对视口滚动固定。
**Why it happens:** CSS 规范规定：sticky 只能相对于最近的滚动容器（scroll container），而 `overflow:hidden` 会创建 BFC 但不是滚动容器，导致 sticky 行为退化为 static。
**How to avoid:** 使用 flex-shrink:0 方案——将 `.pal-sticky-bottom` 设为 `flex-shrink:0`，将可滚动内容包裹在 `flex:1; overflow-y:auto` 的 `.pal-scroll-area` div 内，`#paletteBody` 设 `overflow:hidden` + 固定高度（或让 `#paletteSection` + `#left-scroll` 共同约束高度）。
**Warning signs:** 底部区随页面滚动消失（不固定）。

### Pitfall 2：`refreshSavedDropdown()` 的 selectedPaletteKey 同步问题

**What goes wrong:** 原有逻辑中，选择下拉列表条目只更新 `selectedPaletteKey` 变量，需要再点「加载」按钮才真正加载色卡。新设计中「选中即加载」，但 `selectedPaletteKey` 模块变量必须同步更新，否则 `savePaletteBtn` 的保存逻辑可能使用旧 key。
**How to avoid:** 在 `div.addEventListener('click', ...)` 内第一行更新 `selectedPaletteKey = name`，再调用 `setCurrentPalette()`。

### Pitfall 3：combobox 下拉 `#savedPaletteOptions` 定位

**What goes wrong:** 原 `.custom-select-options` 的 CSS 是 `position:absolute; top:calc(100%+4px); left:0; right:0`，依赖父容器为 `position:relative` 的 `.custom-select`。新布局中如果去掉 `.custom-select` 包裹层，options div 会错位。
**How to avoid:** 保留 `.custom-select` wrapper div 包裹 input+dropBtn+saveBtn 行和 options div，或者手动给 `#savedPaletteOptions` 设置 `position:absolute; top:100%; left:0; right:0` 并给其父容器设 `position:relative`。

### Pitfall 4：`highlightMatchingSwatches()` 遍历包含 `.add-swatch`

**What goes wrong:** `highlightMatchingSwatches()` 用 `document.querySelectorAll('#swatchesGrid .swatch')` 选择节点（第 876 行），并按 idx 对比 `currentPalette[idx]`。如果 `.add-swatch` 也被 `.swatch` 选择到（如果 class 命名不当），idx 会溢出 currentPalette 范围导致 undefined 错误。
**How to avoid:** `.add-swatch` 的 CSS class 不包含 `.swatch`，querySelector 用 `.swatch`（精确类名匹配），确保 `.add-swatch` 不被选中。当前 `highlightMatchingSwatches` 已有 `if (idx >= currentPalette.length) return` 保护（第 877 行），但最好从根本上保证类名不重叠。

### Pitfall 5：删除 tab 结构时遗漏 JS 清理

**What goes wrong:** 第 2641–2649 行的 tab 切换 JS 在 tab-btn 元素删除后会因 `querySelectorAll('.tab-btn')` 返回空 NodeList 而静默，不报错但留下无用代码。
**How to avoid:** 同时删除 JS 中的 tab 事件绑定块（第 2641–2649 行）。

### Pitfall 6：`uploadPaletteFile` 中对 `#palFilename` 的引用

**What goes wrong:** `uploadPaletteFile()` 函数（第 1805 行）在上传成功后更新 `document.getElementById('palFilename').textContent`。新布局中 `#palFilename` 元素（位于原 `#tab-upload` 内）被删除，会引发 null 引用（`if (palFilenameEl)` 已有保护，不报错但更新无效）。
**How to avoid:** 上传成功时改用 `palShowStatus()` 输出文件名，不依赖 `#palFilename` DOM 元素；或将 `#palFilename` 保留为 `display:none` 的隐藏元素。

## State of the Art

| 旧方案 | 新方案 | 原因 |
|--------|--------|------|
| Toggle switch (`#paletteEnabled`) | 无开关（面板始终可用） | 开关语义在移除预览面板后失去意义 |
| Tab 结构（生成/上传 两页） | 线性布局（上传入口下沉至 combobox 下拉） | 减少切换层次，上传是低频操作 |
| Radio card 映射模式（3 张卡片） | `<select>` 下拉框 | 节省垂直空间，视觉层次更清晰 |
| 独立加载 + 删除按钮行 | 下拉选择即加载，删除功能延后 | 操作链路缩短 |
| 导出独立 section（三按钮横排） | 下拉菜单合并到色卡标题栏 | 去掉视觉噪声 |
| `+ 添加颜色` 全宽按钮 | `.add-swatch` 色块（与 swatch 同尺寸） | 视觉一致，节省纵向空间 |

## Open Questions

1. **`#savedPaletteOptions` 定位方式**
   - What we know: 原 `.custom-select-options` 用 `position:absolute`，依赖 `.custom-select` 父容器的 `position:relative`
   - What's unclear: 新 combobox 行是否保留 `.custom-select` wrapper，或者直接用 `position:relative` div 包裹整个 combobox 区块
   - Recommendation: 外层加一个 `position:relative` div 包住 combobox-row 和 options div 即可，不需要 `.custom-select` class（避免 CSS 覆盖混乱）

2. **`#paletteBody` 最大高度控制**
   - What we know: `#left-scroll` 已有 `flex:1; overflow-y:auto`；`#paletteBody` 当前 `display:flex; flex-direction:column; gap:10px`
   - What's unclear: `.pal-scroll-area` 是否需要 `max-height` 硬编码，还是依赖父容器自然约束
   - Recommendation: `.palette-section` 改 `overflow:hidden → overflow:visible`；`.palette-body` 改为 `display:flex; flex-direction:column; max-height:50vh; overflow:hidden`（让 `.pal-scroll-area` 吸收溢出）。实际高度由 `#left-scroll` 决定，50vh 作为合理上限保证底部区域始终可见。
