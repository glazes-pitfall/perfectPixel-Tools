## Summary

Phase 5.1 fixes three tightly linked visual bugs all rooted in a single DPR misapplication in the selection canvas rendering layer. The `selCtx` (selection-canvas context) already has a permanent `scale(dpr, dpr)` applied at init time in `initCanvases()`. Both `drawAnts()` and `_marqueeDrawPreview()` incorrectly wrap their drawing calls in an additional `save() / scale(dpr, dpr) / restore()`, causing every drawn path to be rendered at DPR² scale (4x on a Retina display). The visually bizarre "slow purple-gray ants" symptom is a direct consequence: 4px-wide strokes from the double DPR scale, combined with the two-color (white + black) double-stroke approach, blend into purple-gray on dark backgrounds.

The fix is surgical: remove the `save/scale/restore` blocks from both locations, fix the `clearRect` calls to use logical pixel dimensions (not physical buffer dimensions), and switch to a single `difference` composite operation stroke so ants are always visible regardless of background color.

**Primary recommendation:** Remove the double DPR wrapping from `drawAnts()` and `_marqueeDrawPreview()`, switch to `globalCompositeOperation = 'difference'` with a single white stroke, and verify at DPR=2 (Retina) that ants appear as crisp 1-logical-pixel lines.

## Architecture Patterns

### Pattern 1: Single DPR Scale Applied at Init — Never Per Draw Call

**What:** The overlay canvases (`selCtx`, `cursorCtx`) receive a one-time permanent `scale(dpr, dpr)` at initialization. All subsequent draw calls use logical pixel coordinates — no further scaling is needed or correct.

**Why it works:** After `ctx.scale(dpr, dpr)`, the context maps `1 logical unit → dpr physical pixels` automatically. Any re-application of `save()/scale()/restore()` before drawing compounds the scale: at DPR=2 the effective scale becomes 4x, making a `lineWidth=1` stroke appear as a 4px-wide line.

**Correct Pattern:**
```javascript
// In initCanvases() — ONE TIME ONLY:
selCtx = selCanvas.getContext('2d');
selCtx.scale(dpr, dpr);   // permanent — all future draws use logical coordinates

// In drawAnts() — NO save/scale/restore wrapper:
function drawAnts() {
  if (!EditorState.selectionMask || !_antsPath) { antsRafId = null; return; }
  selCtx.clearRect(0, 0, EditorState.width, EditorState.height);  // logical coords
  selCtx.lineWidth = 1;
  selCtx.setLineDash([4, 4]);
  selCtx.globalCompositeOperation = 'difference';
  selCtx.strokeStyle = '#ffffff';
  selCtx.lineDashOffset = antsDashOffset;
  selCtx.stroke(_antsPath);
  selCtx.globalCompositeOperation = 'source-over';  // reset
  antsDashOffset = (antsDashOffset - 1 + 8) % 8;
  antsRafId = requestAnimationFrame(drawAnts);
}
```

**Anti-pattern (current buggy code):**
```javascript
// WRONG — selCtx already has permanent scale from initCanvases:
selCtx.clearRect(0, 0, selCanvas.width, selCanvas.height);  // BUG: physical dims
selCtx.save();
selCtx.scale(dpr, dpr);   // BUG: double scaling — effective scale = dpr²
selCtx.lineWidth = 1;     // renders as 4px wide on DPR=2 screen
// ... draw ...
selCtx.restore();
```

### Pattern 2: Inverse-Color Rendering with `globalCompositeOperation = 'difference'`

**What:** The `'difference'` composite operation computes `|existing_pixel - new_pixel|` per channel. Drawing a white (`#ffffff`) stroke over any background color produces the mathematical complement: white on black → white, white on white → black, white on gray → visible contrast. This is exactly the behavior needed for selection ants to be visible on any background.

**Mathematical guarantee:** For any background channel value B and drawn channel D=255 (white):
- Result = |B - 255| = 255 - B (since B ≤ 255)
- Dark pixel (B near 0): result near 255 → bright/visible
- Light pixel (B near 255): result near 0 → dark/visible

**Correct Pattern:**
```javascript
// Before stroking ants:
selCtx.globalCompositeOperation = 'difference';
selCtx.strokeStyle = '#ffffff';
selCtx.lineDashOffset = antsDashOffset;
selCtx.stroke(_antsPath);
// CRITICAL: reset after drawing so other canvas operations use normal compositing
selCtx.globalCompositeOperation = 'source-over';
```

**Why reset matters:** If `difference` is left active, subsequent canvas operations (clearRect and future stroke calls) will use the wrong composite operation. Always reset to `'source-over'` immediately after the ants stroke.

### Pattern 3: `clearRect` Must Use Logical Coordinates

**What:** After `ctx.scale(dpr, dpr)`, `clearRect(0, 0, w, h)` expects `w` and `h` in logical pixels. Using physical buffer dimensions (`selCanvas.width`, `selCanvas.height` which are `imageWidth * dpr`) causes the `clearRect` to clear `dpr²` times the needed area — not harmful per se, but semantically wrong and confusing.

**Correct:**
```javascript
selCtx.clearRect(0, 0, EditorState.width, EditorState.height);  // logical
```

**Incorrect:**
```javascript
selCtx.clearRect(0, 0, selCanvas.width, selCanvas.height);  // physical buffer dims
```

### Pattern 4: Marquee Drag Preview Uses Same Fix

**What:** `_marqueeDrawPreview(rx, ry, rw, rh)` has the identical DPR double-scale bug. The same fix applies: remove `save/scale/restore`, use logical clearRect, and apply `difference` composite for the drag-preview rectangle.

**Note on drag preview:** The marquee drag preview uses `strokeRect` rather than `stroke(path2D)`. The fix pattern is the same:

```javascript
function _marqueeDrawPreview(rx, ry, rw, rh) {
  selCtx.clearRect(0, 0, EditorState.width, EditorState.height);  // logical
  selCtx.lineWidth = 1;
  selCtx.setLineDash([4, 4]);
  selCtx.globalCompositeOperation = 'difference';
  selCtx.strokeStyle = '#ffffff';
  selCtx.lineDashOffset = 0;
  selCtx.strokeRect(rx + 0.5, ry + 0.5, rw, rh);
  selCtx.globalCompositeOperation = 'source-over';  // reset
}
```

### Anti-Patterns to Avoid

- **Re-applying `scale(dpr, dpr)` inside draw functions:** Since `initCanvases()` already applies this permanently, any per-draw `save/scale/restore` block compounds the scale. On DPR=2 this makes `lineWidth=1` render as a 4px stroke.
- **Using physical canvas dimensions in clearRect after DPR scale:** After `ctx.scale(dpr, dpr)`, coordinates are in logical pixels. `selCanvas.width` and `selCanvas.height` are physical pixels (= image_size × DPR). Using them in `clearRect` passes values DPR× too large.
- **Leaving `globalCompositeOperation = 'difference'` active:** Forgetting to reset to `'source-over'` after the stroke will cause subsequent canvas operations to use difference compositing, producing unexpected results.
- **Assuming `difference` on a transparent background works intuitively:** If the underlying canvas layer is transparent (alpha=0), `difference` on `rgba(255,255,255,1)` still produces white — this is correct. However if the `globalAlpha` is not 1, results may differ.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Always-visible selection outline on any bg | Custom dual-stroke (white+black offset) | `globalCompositeOperation = 'difference'` single white stroke | Built into Canvas 2D; handles all background colors mathematically; simpler code |
| HiDPI canvas rendering | Per-draw-call DPR re-scaling | One-time `ctx.scale(dpr, dpr)` at init | Canvas 2D context transform is cumulative; re-applying creates compounding scale errors |

**Key insight:** The Canvas 2D context transform matrix is persistent and cumulative. The correct DPR pattern is: scale ONCE at init, then treat all drawing coordinates as logical pixels.

## Common Pitfalls

### Pitfall 1: DPR Double Application — The Root Bug

**What goes wrong:** `drawAnts()` and `_marqueeDrawPreview()` both apply `save(); scale(dpr, dpr); ...; restore()` around their drawing code. Since `initCanvases()` already applied `selCtx.scale(dpr, dpr)` permanently, this second application effectively multiplies the scale again. On a Retina display (DPR=2), the path renders at 4x scale, making `lineWidth=1` appear as a 4px-wide stroke.

**Why it happens:** The `save/scale/restore` pattern is the correct way to temporarily apply DPR when the context has NOT been pre-scaled. But when the context already has a permanent scale (as the project's architecture requires), using this pattern per-draw-call is incorrect.

**How to avoid:** Never put `scale(dpr, dpr)` inside draw functions. Scan for any draw function that does `save(); scale(dpr, dpr); ...; restore()` and remove the save/scale/restore wrapper.

**Warning signs:** Strokes appearing 4x thicker than expected on Retina; visual artifacts with double-stroke colors blending (color mixing from overlapping thick strokes).

### Pitfall 2: Forgetting to Reset `globalCompositeOperation`

**What goes wrong:** After setting `selCtx.globalCompositeOperation = 'difference'` for the ants stroke, failing to reset to `'source-over'` means subsequent calls to `selCtx.clearRect()` or any other canvas operations on `selCtx` will use difference compositing — which may produce unexpected transparency or coloring.

**Why it happens:** `globalCompositeOperation` is a persistent state on the context, like `strokeStyle` or `lineWidth`. Setting it for one operation affects all subsequent operations unless reset.

**How to avoid:** Immediately after `selCtx.stroke(_antsPath)`, add `selCtx.globalCompositeOperation = 'source-over'`.

### Pitfall 3: `clearRect` Physical vs Logical Coordinates

**What goes wrong:** Using `selCanvas.width` and `selCanvas.height` (physical buffer dimensions = image_size × DPR) in `clearRect()` after the context has been pre-scaled by DPR. This passes values that are DPR× larger than the logical canvas area.

**Why it happens:** `selCanvas.width` is the backing store width in physical pixels. After `scale(dpr, dpr)`, the coordinate system is in logical pixels. The mismatch is easy to miss because `clearRect` with oversized dimensions still clears the canvas (just covers more area than needed), which masks the bug.

**How to avoid:** Always use `EditorState.width` and `EditorState.height` (logical image dimensions) in `clearRect` when drawing on the pre-scaled overlay canvases.

### Pitfall 4: The Purple-Gray Visual Symptom is a Consequence, Not a Separate Bug

**What goes wrong:** Developer sees "purple-gray ants" and investigates as a color problem, missing the root cause.

**Why it happens:** At DPR=2, the double-scale makes `lineWidth=1` render as 4px. The white and black double-strokes (offset by 4 dash units) create thick overlapping stripes. At sub-pixel boundaries on dark backgrounds, the rendering engine anti-aliases the mix of black and white strokes into gray/purple.

**How to avoid:** This symptom disappears automatically once the DPR double-application is fixed. No separate color fix is needed.

## Open Questions

1. **`difference` composite and transparent `selCanvas` background**
   - What we know: `difference` computes `|top - bottom|` per channel. The selection canvas sits above the pixel canvas. The browser composites the full stack visually.
   - What's unclear: `globalCompositeOperation` operates on the canvas's OWN buffer, not the visual composite. The selection canvas has a transparent background. Drawing white with `difference` over a transparent canvas yields white (|255-0|=255 for RGB, but alpha behavior needs verification).
   - Recommendation: After implementing, do a visual test on both dark and light pixel art backgrounds. If ants appear white-on-dark AND dark-on-light, the behavior is correct. If ants appear uniformly white regardless of background, the `difference` operation is compositing against the selection canvas's own transparent buffer (not the pixel canvas below) — in that case, an alternative approach using two alternating dash colors without compositing may be needed. **This is the one risk area in the implementation.**

2. **`lineWidth = 1` after DPR fix: exact rendered width**
   - What we know: After removing the double-DPR, `lineWidth=1` should render as 1 logical pixel = `dpr` physical pixels (crisp on Retina, as intended).
   - What's unclear: Path2D coordinates in `rebuildAntsPath` use integer pixel coordinates. The `0.5` pixel offset used in `_marqueeDrawPreview` (`strokeRect(rx + 0.5, ...)`) avoids pixel boundary blurring for straight lines. `rebuildAntsPath` does NOT use `+0.5` offsets — this may cause 2-physical-pixel-wide lines on Retina due to sub-pixel positioning.
   - Recommendation: After fixing the DPR bug, visually verify `drawAnts` line sharpness at DPR=2. If blurry, consider adding `0.5` offsets to the `moveTo`/`lineTo` calls in `rebuildAntsPath`. This is within Claude's discretion per CONTEXT.md.
